jMax 3 development road map

jMax 3 should be released as soon as possible. The developments mentioned in this paper are considered as urgent and have to be completed to be included to this release.

The release of jMax 3 can be split into two steps: First a user release which provides the user with an interface (modules and syntax) introducing the dynamic client/server connection, a user friendly configuration and launcher applications embedded to the different platforms supported by the release:
  • Linux
  • Windows
  • MacOs X
  • SGI
The implementation of a package for soundfile access on all four platforms is to be considered as critical for this release.

portable soundfile i/o

The reading and writing of soundfiles must be handles by a platform dependent package and implemented for all four platforms.

dynamic client/server connection

This is the most important group of new features for jMax 3:
  • the server can run without the client
  • the client can be connected, deconnected and reconnected to the server at any moment
  • multiple clients can be connected to the server

project, package & configuration

A graphical editor should be provided for the editing of packages/projects.
How to handle the configuration???

Default midiport and audioport becomes a configuration field instead of global variable.

variables and expression parsing

The current expression parsing and mechanism to resolve variable names are major sources of instability and annoyance. jMax 3 should include a very first version of a new parser providing at least a very simple syntax for messages close to the current jMax (MAX) syntax for messages and simple expressions.
  • parsing of strings to FTS' internal representations is done by the server
  • the client/server protocol permits to send parsed message and atoms as well as unparsed strings
The parser has to be compatible with the future developments of jMax (implemented with a minimal effort!!). The parser should try to keep a maximum compatibility with MAX style messages and the expression syntax of jMax.

The parser should introduce the possibility to specify the package of class at the instantiation of an object. This way the transition towards a new object set can be simplified.


platform embedding

At least for MacOsX and Windows version but as well for Linux a jMax launcher application should be provided, which fully integrates into the given OS in a way that opening of jMax documents such as patches and projects can be done in the standard way provided by a given OS (double-click, drag and drop, etc.).

message system

For the first user release the changes of the message system should be minimal. Nevertheless some changes has to be done in order to support the new parser and client server communication. The outlines of the message system for FTS
after the jMax 3 user release should be developed in advance.

GUI features

Editors should be provided for fvec, messtab (maybe to be renamed to "dictionary" or so?), and the MIDI track.

multi-thread

Several threads running their own scheduler and communicating via message fifo (i.e. via control only)

Do an async sound file reader (based on methods callback) and reimplement Direct-to-Disk using async sound file reader.
