Disclamier

The current version of jMax use TCL as its scripting language, and the feature described in this document is currently based on TCL; future version of jMax may use a different scripting language, so the features described here will change, will be implemented with a different language and/or syntax, or will simply vanish.

Introduction

The system configuration is based on system parameters that can be set either by setting a TCL variable in the .jmaxrc file or giving a command line argument when running jMax.

Since a patch can be used on different platforms, platform specific information should not be included in the patches themselves, but specified in site, user or project configuration files.

For each host type supported by jMax a set of i/o devices is defined, which can be used with these commands. Refer to the index of supported host types to see which audio devices are provided for each of them.

Host Parameters

Since jMax has a client/server architecture the host parameters are to be set in order to specify the host on which the jMax server should be run. By default this is the local host where jMax is launched (and so the client is started up).
In this case the host type must be specified using the jmaxHostType parameter.

When the parameter jmaxHost is given, the server will run on the named remote host.
In this case jMax automaticly should find the host type in the host table of the site specific configuration file (so that is doesn't has to be explicitly set by the user when running jMax).

The parameter jmaxArch is automaticly set for a specific host type so it should not be set by the user at all!

jmaxHost

Specifies the host where to run the server. The value can be either local (default) to specify the host where the client run, or a complete canonical internet name for an other host, like mymachine.mycompany.com.
Beware that jMax currently uses the rsh Unix command to run the server remotely, so in order to work you must set your .rhosts file correctly (see the rsh and rhosts man page for details).

jmaxHostType

Sets the type of the host running the server.
The host type determines the hardware architecture and the implementation of the i/o devices.
See the index of supported host types.

jmaxArch

Sets the hardware architecture of the host running the server.
It is used to choose the proper binaries for the given architecture (do not set).

Miscellaneous Parameters

jmaxConnection

Choose the kind of connection between the Java client and the server; Supported values are currently udp (default) for an UDP based connection, and client for a TCP based connection. UDP will give a better real time response for controllers. Use TCP only if your network is complex and you need the reialability of a TCP connection.

jmaxSplashScreen

If this value is hide the splash screen will not be shown during startup.
The deafault value is show.

jmaxFastFileBox

If the value is true, the file selection box used by jMax will not shown specific icons for known file types, and will always show all the files in a directory, and not only the files of a known file type, so it's use will be substantially faster.
Depending on your installation, the default speed can be ok, or you may need to specify this option.

jmaxDefaultFont

Specify the default font used in the patch editor.
Default to SansSerif.

Audio Configuration

The following set of commands handles the audio configuration. The first two of them are used to set the audio configuration parameters and should be used before the others.

jmaxSetSampleRate sr

Set the jMax global sampling rate.
(Note: For now, the user himself must keep the sampling rate given for jMax consistent with the settings of the physical device using the utillities provided by the operating system, like the SGI audiopanel application.)

jmaxSetAudioBuffer buf

Set the length of the audio buffer used, i.e. the length, in samples, of the output FIFO used to compensate the variation of load in jMax. Typical values are between 256 and 1024, can be more or less depending on the application.
Lower values cause lower i/o latencies, while higher values reduce possible audio clicks caused by system overloads.

openDefaultAudioIn device

Open the audio input device device as the default audio device, i.e. the device used by the adc~ object.

openAudioIn name device

Open a input device named name connected to the objects in~ which have name as first argument (allowing the use of multiple input devices - one for each object).

resetAudioIn

Reset the effects of the previous commands, i.e. close all the input audio devices, and delete the registered names (usefull in projects files or other situation where a clean input configuration must be installed from scratch).

openDefaultAudioOut device

Open the audio output device device as the default audio device, i.e. the device used by the dac~ objects.

openAudioOut name device

Open an output device named name connected to the objects out~ that have name as first argument (allowing the use of multiple output devices - one for each object).

resetAudioOut

Reset the effects of the previous commands, i.e. close all the output audio devices, and delete the registered names (usefull in projects files or other situation where a clean output configuration must be installed from scratch).

MIDI Configuration

These commands handle the MIDI configuration:

openDefaultMidi device

Open the MIDI device device as the default MIDI port, i.e. the device used by all the MIDI objects without a port number argument.

openMidi num device

Open the MIDI device device as the MIDI port installed on port number num i.e. the device used by all the MIDI objects with a port number argument of num. A device installed with port number of 0 will be used by all the MIDI related object without a MIDI port number.

resetMidi

Reset the effects of the previous commands, i.e. close all the MIDI devices, and delete the registered port number (usefull in projects files or other situation where a clean MIDI configuration must be installed from scratch).

Using ALSA

The Linux ALSA sound drivers are supported by a dynamically loaded package, named alsadev-0.4. The currently supported ALSA version is 0.4.

To use jMax with the Linux ALSA sound drivers, jMax must be compiled with ALSA support (see Compiling jMax for further information. The ALSA package can then be loaded by adding the following line in your .jmaxrc file, inside the start hook: 
	package require alsadev-0.4

The alsadev package defines the following devices: alsaStereoIn for audio input, alsaStereoOutput for audio output. These devices can be used by adding the following line in your .jmaxrc file, inside a start hook: 
	openDefaultAudioIn alsaStereoIn
	openDefaultAudioOut alsaStereoOut

Below is an example of a .jmaxrc file for using ALSA audio input and ouput: 
when start {
	package require alsadev-0.4
	jmaxSetSampleRate 44100
	jmaxSetAudioBuffer 2048

	openDefaultAudioIn alsaStereoIn
	openDefaultAudioOut alsaStereoOut
}
